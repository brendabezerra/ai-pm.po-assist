# /qa — QA Testing and Bug Tracking

## What This Command Does

Takes a completed issue (after implementation) and generates a full QA test story, then guides the PM through the test cycle. Creates bug reports for any failures found. Tracks the pass/fail status of each test scenario until the issue is fully verified.

This covers steps 8–9 of the PO workflow: **test implementation** and **track bugs**.

---

## When to Use

- After a developer marks an issue as implemented and ready for QA
- To generate QA test stories from a dev plan (if not already generated by `/devplan`)
- To open bug reports for failures found during testing
- To close an issue's QA cycle after all scenarios pass

---

## How to Run

```
/qa
```

The agent will ask:
1. Which project?
2. Path to issue file or dev plan
3. Is implementation complete and deployed to test environment? (confirm before starting)

---

## Execution Steps

### Phase 1 — Generate QA Story

1. **Read the issue** — acceptance criteria, definition of done, linked SPEC FEAT
2. **Read the QA story stub** in `projects/<name>/docs/stories/` (if generated by `/devplan`)
   - If stub exists: complete it (fill in test scenarios, test data, regression impact)
   - If no stub: generate from scratch using `skills/docs/prompts/story_writer.md`
3. **Build test scenarios** — one per acceptance criterion, plus:
   - At least one happy path scenario
   - At least one edge case scenario
   - At least one error/failure scenario
4. **Present** the full QA story to the user for review before testing begins

### Phase 2 — Run Test Cycle

5. **Walk through each test scenario** — present one at a time:
   - Show: Given / When / Then
   - Ask: **Pass or Fail?**
   - If pass: mark ✓ and move to next
   - If fail: proceed to Phase 3 (bug report)

### Phase 3 — Bug Reporting (on failure)

6. **Open a bug report** using `templates/bug_report.md`:
   - Pre-fill: project, linked issue, linked QA story, failing test scenario
   - Ask user for: actual behavior, steps to reproduce, environment
   - Assign severity: P1 (blocker — stops QA), P2 (major), P3 (minor)
7. **Save** bug report to `projects/<name>/docs/bugs/YYYY-MM-DD-bug-[slug].md`
8. **Link** bug report back to the issue file
9. **Ask**: Continue testing other scenarios, or pause for fix?
   - Continue: resume test cycle, mark this scenario as failed
   - Pause: summarize open bugs, suggest next step is fix cycle

### Phase 4 — QA Closure

10. Once all scenarios are passed (or all bugs are fixed and verified):
    - Mark QA story status as `passed`
    - Update issue DoD checklist: ✓ QA test scenarios passed
    - Ask user: "Ready to proceed to release checklist?"
    - If yes: generate `templates/release_checklist.md` pre-filled for this issue

---

## Output Files

| Document | Template | Save Location |
|---|---|---|
| QA Story | `templates/qa_story.md` | `projects/<name>/docs/stories/YYYY-MM-DD-qa-[issue].md` |
| Bug Report(s) | `templates/bug_report.md` | `projects/<name>/docs/bugs/YYYY-MM-DD-bug-[slug].md` |
| Release Checklist | `templates/release_checklist.md` | `projects/<name>/docs/YYYY-MM-DD-release-[feature].md` |

---

## Next Steps After This Command

| Outcome | Next Step |
|---|---|
| All scenarios pass | Generate release checklist → push to dev branch |
| P1 bug found | Pause QA, developer fixes, re-run `/qa` for affected scenario |
| P2/P3 bugs found | Log, continue QA, address bugs before release checklist |

---

## Guardrails

- Never mark an issue as QA-passed if any scenario has failed without a verified fix
- All P1 bugs must be resolved before proceeding to the release checklist
- Bug reports must be linked to the issue and the failing QA scenario
- Test scenarios must include edge cases — do not test only the happy path
- Never fabricate test results — only mark scenarios as passed when user confirms
