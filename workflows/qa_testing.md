# QA Testing Workflow

## Overview

How a completed implementation is tested, bugs are tracked, and an issue is closed for release. Covers steps 8–9 of the PO workflow.

---

## Steps

```
Developer marks issue as implemented
            ↓
      /qa — generate full QA story
            ↓
      Run test scenarios (pass / fail)
            ↓
        ┌───┴───┐
      Pass     Fail
        │         ↓
        │    Bug report created
        │         ↓
        │    Developer fixes
        │         ↓
        │    Re-test failed scenarios
        │         ↓
        └───┬───┘
            ↓
      All scenarios pass
            ↓
      Release checklist generated
            ↓
      Push to development branch
```

---

## Step 1 — Confirm Implementation is Ready

**Trigger:** Developer communicates that the issue is implemented.

**Check before starting QA:**
- [ ] Code is deployed to the test environment
- [ ] All implementation steps in the dev plan are marked complete
- [ ] Backend story acceptance criteria are met (developer confirms)
- [ ] No known P1 blockers in the PR

**If any check fails:** Do not start QA. Return the issue to the developer.

---

## Step 2 — Generate Full QA Story

**Action:** Run `/qa` with the issue file path.

The agent will:
1. Read the issue, dev plan, and linked SPEC FEAT section
2. Complete the QA story stub (if generated by `/devplan`) OR generate from scratch
3. Build test scenarios: one per acceptance criterion + edge cases + error scenarios
4. Present the full QA story for PM review before testing begins

**Quality bar for QA story:**
- [ ] Happy path: at least one scenario
- [ ] Edge cases: at least one scenario
- [ ] Error / failure scenarios: at least one scenario
- [ ] Every acceptance criterion from the issue has a corresponding test scenario

**Save to:** `projects/<name>/docs/stories/YYYY-MM-DD-qa-[issue].md`

---

## Step 3 — Run Test Scenarios

**Action:** Walk through each test scenario in the QA story.

For each scenario:
1. Reproduce the Given / When condition
2. Observe the actual outcome
3. Mark: **Pass** or **Fail**

**If Pass:** Continue to next scenario.

**If Fail:** Proceed to Step 4 before continuing.

---

## Step 4 — Log Bug Report (on failure)

**Trigger:** Any test scenario fails.

**Action:** Run `/qa` bug mode (agent automatically prompts for bug details when a scenario fails).

**The agent creates** `templates/bug_report.md` pre-filled with:
- Linked issue and QA story
- Failing test scenario
- Ask for: actual behavior, steps to reproduce, environment

**Severity assignment:**
- **P1 (Blocker):** Prevents completing the test cycle or breaks a core user flow
- **P2 (Major):** Significant deviation from expected behavior, workaround exists
- **P3 (Minor):** Small visual or non-critical functional issue

**Save to:** `projects/<name>/docs/bugs/YYYY-MM-DD-bug-[slug].md`

**Update:** Issue file — add bug report to Linked Documents section.

**Decision after P1 bug:**
- Pause QA entirely, assign fix to developer
- Resume other test scenarios only if they are not affected by the P1 bug

**Decision after P2/P3 bug:**
- Log the bug, continue remaining test scenarios
- Resolve before generating release checklist

---

## Step 5 — Fix and Re-test Cycle

**Trigger:** Developer completes a bug fix.

**Action:**
1. Confirm fix is deployed to test environment
2. Re-run only the **failed scenarios** from the QA story (not the full story)
3. If scenario now passes: mark ✓, update bug report status to `resolved`
4. If scenario still fails: update bug report with new findings, reassign to developer

**Repeat until all failed scenarios pass.**

---

## Step 6 — Close QA Cycle

**Trigger:** All test scenarios marked as passed.

**Action:**
1. Update QA story status to `passed`
2. Update issue DoD checklist: ✓ QA test scenarios passed, ✓ No open P1 bugs
3. Update all P2/P3 bug reports to `resolved` (or accept as known issues with PM sign-off)
4. Generate `templates/release_checklist.md` pre-filled for this issue

**Release checklist confirms:**
- [ ] All issues closed
- [ ] All QA stories passed
- [ ] No open P1 bugs
- [ ] PR reviewed and approved
- [ ] Documentation updated
- [ ] Stakeholder sign-off

---

## Step 7 — Authorize Push to Development Branch

**Trigger:** Release checklist is fully checked.

**Action:**
1. PM signs off on release checklist
2. Developer pushes to development branch
3. Update issue status to `done`
4. Update `projects/<name>/context.md` sprint goals to reflect completion

---

## Data Flow

| Input | Source | Output | Destination |
|---|---|---|---|
| Issue + dev plan | `projects/<name>/docs/issues/` | QA story | `projects/<name>/docs/stories/` |
| Failed test scenario | Manual testing | Bug report | `projects/<name>/docs/bugs/` |
| All scenarios passed | Manual testing | Release checklist | `projects/<name>/docs/` |

---

## Dependencies

- **Test environment** — must be available before QA starts
- **Dev plan + issue** — must be complete before QA story is generated
- **SPEC** — provides acceptance criteria that QA scenarios are derived from

---

## Quality Check

An issue is ready for release only when:
- [ ] Every acceptance criterion in the issue has a QA test scenario
- [ ] Every test scenario is marked passed
- [ ] Zero open P1 bugs
- [ ] All P2/P3 bugs resolved or accepted by PM with documented rationale
- [ ] Release checklist completed and signed by PM
